# ADR-001: Framework and stack for the Scenario Creator

## Status
Accepted.

## Context
The app is a form-heavy, multi-panel editor: 11 editor panels (6 base, 5
advanced), each a schema-driven form with numeric ranges, enums, toggles,
repeating lists (events, goals), and a live preview. Two hard architectural
constraints shape the choice:

1. **Theme abstraction** — every asset, icon, color token, label string and
   style resolves through one theme registry backed by `/assets/<theme>/`
   folders. Swapping the folder must reskin the app with zero component-code
   changes.
2. **Domain abstraction** — the editor structure is data (a blueprint derived
   from the wiki research plus a declarative domain schema), not hardcoded
   panels. Components must render *whatever* the blueprint/domain says.

Tests must prove theme-swap structural equivalence and JSON round-trip safety
without weakening, which pushes toward a stack where components can be
rendered and compared as plain strings in Node.

## Decision
**React 19 + Vite 8 + TypeScript (strict) + Zod 4 + Vitest 4.**

- **React** — schema-driven forms are a natural fit for React's declarative
  model: one generic `EditorPanel` maps `FieldSpec[]` to inputs, so adding an
  editor or field is a data change, not a component change. Context cleanly
  injects the theme registry and domain schema, which is exactly the
  seam both abstraction layers need. `react-dom/server`'s
  `renderToStaticMarkup` lets the test suite render the whole app under both
  themes in Node (no browser, no jsdom) and diff the structure — the
  strongest cheap proof that a theme swap changes no component behavior.
- **Vite** — `import.meta.glob` gives the theme registry filesystem-driven
  discovery of `/assets/*/theme.json` and `/assets/*/icons/*.svg`: dropping a
  new theme folder in is genuinely all it takes. Fast dev server for a
  many-panel editor.
- **Vitest** — runs through Vite's transform pipeline, so the same
  `import.meta.glob` theme discovery works identically in tests and app code.
  No mocking of the asset layer, so the tests exercise the real registry.
- **Zod** — wiki-documented ranges (`[0-100]` sliders, enums, ADD/SET
  semantics) become runtime validators *and* inferred static types from one
  declaration. The scenario import path and every editor field share the same
  schema objects, so import validation can't drift from UI validation.
- **TypeScript strict** (plus `noUncheckedIndexedAccess`) — required by the
  spec; also load-bearing for the registry code, which is full of indexed
  lookups into theme/domain maps.

## Alternatives considered
- **Svelte 5** — excellent for forms, but its compiled component model makes
  "render the full app twice and diff markup in Node" less direct, and the
  ecosystem around headless schema-driven form rendering is thinner.
- **Angular** — strong forms story (reactive forms), but far heavier
  scaffolding for a single-page tool, and the DI machinery duplicates what a
  React context provides here in a few lines.
- **Vanilla TS + lit-html** — smallest footprint, but hand-rolling list
  fields, validation display, and re-render orchestration for 11 panels is
  exactly the undifferentiated work a declarative library removes.
- **Form libraries (react-hook-form, TanStack Form)** — deliberately skipped;
  the blueprint-driven renderer keeps state in one plain reducer keyed by
  `editorId.fieldId`, and a form library's own schema layer would compete
  with the domain blueprint as the source of truth.

## Consequences
- Components never import theme or domain files directly; they consume
  `useTheme()` / props only. Enforced by the structural-equivalence test.
- All validation flows through the Zod scenario schema built from the
  blueprint; the import path and the UI cannot diverge.
- No jsdom/browser dependency in tests keeps CI to `npm ci && npm test`.
